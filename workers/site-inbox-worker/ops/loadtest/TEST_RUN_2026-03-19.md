# k6 production-like runs — 2026-03-19

Target: Cloudflare Worker production endpoint (URL withheld), LITE_MODE=1, rate-limit defaults 300/60s per IP, secrets from vault.

## Run 1 — lite profile (10 rps inbox, 5 rps notify)
- Command: `docker run ... grafana/k6 run ops/loadtest/k6-worker-lite.js ...`
- Results: 899 requests, checks 898/899 passed, one inbox 429 (expected). p95 latency 268 ms, max 747 ms. `http_req_failed` ~98% because 429 counted as failures. Confirms rate-limit protection works at this load.

## Run 2 — lite profile with global cap `--rps 5`
- Command: `docker run ... grafana/k6 run --rps 5 ops/loadtest/k6-worker-lite.js ...`
- Results: 329 requests, all checks passed; many iterations dropped and `http_req_failed` ~94% as k6 throttled below scenario rate, still seeing 429s from server. p95 latency 216 ms. Indicates to get a “green” threshold either lower scenario rates (e.g., 3 rps inbox / 1 rps notify) or raise server rate-limits.

Notes:
- No functional errors or 5xx observed; only rate-limit rejections.
- Health/metrics endpoints were verified separately and require bearer auth.
- For CI-friendly green runs: adjust k6 rates to stay under 5 rps/IP or increase RATE_LIMIT_MAX in env when appropriate.
